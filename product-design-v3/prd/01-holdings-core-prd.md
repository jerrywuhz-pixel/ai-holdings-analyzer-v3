# 持仓核心产品 PRD

## 背景

AI 持仓系统 3.0 的登录后 WebApp 不是营销站点，而是用户进入系统后的结构化操作主场。当前阶段最重要的用户可见能力，不是泛化聊天，而是围绕“我持有什么、这些持仓是否正常、今天要处理什么、哪些动作需要确认”形成闭环。

登录前允许新增公开产品站点，用于传播、注册转化、新用户理解和会员权益展示。该站点采用“长期资产管家 + AI 投研任务流”的 B+C 方向；它负责解释产品价值，不改变本 PRD 对登录后持仓核心工作台的范围定义。

本 PRD 只覆盖持仓核心产品范围：Dashboard 持仓摘要、持仓工作台、多 `portfolio_view`、股票 / ETF 详情、Sell Put 工作台、确认中心联动，以及与这些页面相关的状态、错误、权限、埋点和验收。

已确认的产品决策如下：

| 决策 | 结论 |
| --- | --- |
| 全局聊天入口 | 无。WebApp 不做独立全局聊天，AI 能力嵌入具体页面和任务上下文 |
| 登录方式 | 使用 Supabase Auth |
| 付费与配额 | P0 不做真实付费配额；登录前站点可以展示会员方案和权益分层 |
| `portfolio_view` | P0 支持多个 `portfolio_view` |
| 持仓主源 | Futu 为主源 |
| 自动下单 | 不做 |

## 目标

1. 让用户在 30 秒内看懂当前资产是否正常，重点风险是什么，今天需要处理什么。
2. 提供一个统一的持仓工作台，支持多个 `portfolio_view`，并清晰区分股票 / ETF 与期权。
3. 让股票 / ETF 详情页和 Sell Put 工作台分别承载各自的分析、风险、策略和确认需求。
4. 让所有高注意力动作都进入确认中心，形成可追溯、可撤回、可审计的用户流程。
5. 把数据新鲜度、来源可信度、对账状态和确认状态显式展示出来，避免用户对持仓真相产生误判。

## 非目标

1. 不做全局聊天式投资助手。
2. 不做自动下单。
3. 不做 P0 付费墙、套餐配额和复杂计费策略。
4. 不把 `portfolio_view` 当成真实账户或复制资产的容器。
5. 不在 P0 覆盖完整研究平台、清仓复盘平台和完整账户管理中心。
6. 不把股票、ETF、期权混成同一套字段和同一套分析逻辑。

## 用户画像 / 场景

### 用户画像

| 画像 | 特点 | 主要诉求 |
| --- | --- | --- |
| 组合管理型用户 | 同时持有多个券商账户或多个策略组合 | 统一查看资产、切换视图、判断整体风险 |
| 股票 / ETF 持有型用户 | 更关注单票盈亏、趋势和纪律 | 快速看详情、设置止盈止损、回看交易过程 |
| Sell Put 使用型用户 | 重点关注现金占用、DTE、IV、assignment 风险 | 判断是否还能卖、是否该 roll、是否要确认 |
| 高注意力操作用户 | 经常导入交易、修正 OCR、处理冲突 | 需要清晰确认流和审计记录 |

### 典型场景

1. 用户早上打开 Dashboard，先看总资产、现金、今日盈亏和待处理项。
2. 用户切换到某个 `portfolio_view`，查看某个账户或策略组合的持仓结构。
3. 用户点击一只股票，查看该标的的持仓摘要、纪律状态、历史动作和建议。
4. 用户进入 Sell Put 工作台，判断现金占用、到期梯队和候选合约是否符合纪律。
5. 用户处理一条 OCR 或交易冲突，在确认中心完成确认或拒绝。

## 核心用户故事

1. 作为用户，我希望打开 Dashboard 就能看到资产摘要和当前风险，不必再去找多个页面。
2. 作为用户，我希望在同一账号下切换多个 `portfolio_view`，分别查看全部资产、美股账户或期权策略账户。
3. 作为用户，我希望在持仓工作台里清楚地区分股票 / ETF 和期权，不被混在一起的信息干扰。
4. 作为用户，我希望点开股票 / ETF 详情后，能看到盈亏、纪律命中、交易时间线和下一步建议。
5. 作为用户，我希望在 Sell Put 工作台里直接看到现金占用、DTE、IV、assignment 风险和候选合约对比。
6. 作为用户，我希望所有交易草稿、OCR 修正、规则 override 和冲突处理都先进入确认中心，再决定是否确认。
7. 作为用户，我希望系统明确告诉我当前数据来自哪里、是否过期、是否已对账，而不是只给我一个数字。

## 功能范围

### 1. Dashboard 持仓摘要

Dashboard 的职责是“先判断状态，再决定下一步”。它应当展示：

| 内容 | 说明 |
| --- | --- |
| 资产摘要 | 总资产、现金、股票市值、期权资金占用、今日盈亏、累计盈亏 |
| 当前 `portfolio_view` | 当前正在查看的视图名称与切换入口 |
| 数据状态 | 主源、最后同步时间、新鲜度、异常提示 |
| 风险摘要 | 仓位集中、单标的暴露、到期风险、纪律冲突 |
| 今日待办 | 待确认、待修正、待处理的持仓事项 |

Dashboard 不承担完整分析页职责，只展示结论和入口。

### 2. 持仓工作台

持仓工作台是持仓功能的核心入口，要求支持多个 `portfolio_view`，并按品种清晰分区展示。

功能要求：

1. 支持 `portfolio_view` 切换，且切换后只改变展示口径，不改变真实资产。
2. 支持股票 / ETF 持仓分区展示。
3. 支持期权持仓分区展示，首期重点是 Sell Put 相关信息。
4. 支持按来源、市场、账户、状态筛选。
5. 支持从列表进入股票 / ETF 详情或 Sell Put 工作台。
6. 支持查看来源、更新时间、新鲜度和对账状态。

### 3. 股票 / ETF 详情

股票 / ETF 详情页用于回答三个问题：

1. 这只标的现在赚亏多少。
2. 风险和纪律冲突在哪里。
3. 下一步是观察、止盈、止损还是加仓。

页面应包含：

| 模块 | 说明 |
| --- | --- |
| 持仓摘要 | 仓位、市值、成本、浮盈亏、来源 |
| 纪律命中 | 规则状态、是否有 override、是否触发提醒 |
| 价格与收益路径 | 价格变化、成本线、当前收益状态 |
| 止盈 / 止损策略 | 当前策略、建议动作、可执行条件 |
| 交易时间线 | 买入、卖出、加仓、修正、确认记录 |
| 页面内分析 | 对今日变化的解释和下一步建议 |

页面允许生成分析和策略草稿，但不直接写交易事实。

### 4. Sell Put 工作台

Sell Put 工作台是期权产品的独立页面，不复用股票详情页。它围绕现金占用、DTE、IV、流动性、愿接股和确认来组织。

页面应包含：

| 模块 | 说明 |
| --- | --- |
| 现金摘要 | 可用现金、已占用现金、剩余空间 |
| 当前持仓 | short put 合约、DTE、delta、IV、风险等级 |
| 到期梯队 | 临近到期、中期、远期分层 |
| 候选合约 | underlying、strike、DTE、premium、风险结果 |
| 交易纪律 | 愿接股规则、财报前限制、现金上限 |
| 确认入口 | 生成交易草稿并进入确认中心 |

Sell Put 工作台可以生成交易草稿，但不能自动下单。

### 5. 确认中心联动

确认中心是高注意力页面，凡是可能改变事实、产生草稿、修正规则或处理冲突的动作，都应进入这里。

确认对象包括：

1. 交易录入。
2. OCR 修正。
3. Sell Put 草稿。
4. 规则 override。
5. 批量导入。
6. 券商、手工、OCR 数据冲突。

确认中心要求：

| 要求 | 说明 |
| --- | --- |
| 结构化预览 | 用户先看清楚对象、来源、风险和动作上限 |
| 显式决策 | 确认、拒绝、退回 |
| 状态联动 | WebApp 和微信看到一致状态 |
| 审计记录 | 保留确认人、时间、来源快照和备注 |

## 页面 / 流程

### 页面结构

| 页面 | 作用 |
| --- | --- |
| Dashboard | 总览资产摘要、风险和待办 |
| 持仓工作台 | 多 `portfolio_view`、股票 / ETF、期权统一入口 |
| 股票 / ETF 详情 | 单标的盈亏、纪律、分析、时间线 |
| Sell Put 工作台 | 期权现金占用、DTE、候选合约与风险 |
| 确认中心 | 高注意力确认、修正、冲突处理 |

### 主流程

1. 登录后进入 Dashboard。
2. 在 Dashboard 查看资产摘要、数据状态和待处理项。
3. 进入持仓工作台，切换 `portfolio_view`。
4. 点击股票 / ETF 进入详情，或点击期权进入 Sell Put 工作台。
5. 对需要确认的动作进入确认中心。
6. 确认后，状态回写到持仓工作台、详情页或 Dashboard 待处理区。

### 首次使用流程

1. 用户通过 Supabase Auth 登录。
2. 系统进入默认 `portfolio_view`。
3. 页面展示 Futu 主源同步结果、数据新鲜度和待确认项。
4. 用户根据摘要进入具体持仓页处理动作。

## 状态与错误

### 关键状态

| 对象 | 状态 | 用户可见表现 |
| --- | --- | --- |
| `portfolio_view` | 默认、可切换、编辑中 | 顶部当前视图和切换器 |
| 持仓数据 | 新鲜、过期、冲突、待确认 | badge、提示和降级展示 |
| 股票 / ETF 持仓 | 正常、纪律命中、需复核 | 颜色和状态标签 |
| 期权持仓 | 正常、高注意、临近到期、需确认 | 风险标签和确认入口 |
| 确认对象 | 待处理、已确认、已拒绝、已失效 | 列表状态同步 |

### 常见错误

| 错误 | 处理原则 |
| --- | --- |
| 数据过期 | 明确标注，降低推荐力度，不生成高风险草稿 |
| 来源冲突 | 进入确认中心，不直接覆盖 |
| 缺少关键字段 | 不展示为可执行建议，只展示信息或提示补全 |
| 视图切换失败 | 保留当前视图并提示重试 |
| 确认提交失败 | 保持待处理状态，避免误以为已生效 |

## 权限 / 合规边界

1. 所有页面都以当前 Supabase Auth 身份为准，数据隔离以账号为边界。
2. 用户只能查看和操作自己账号下的数据与视图。
3. `portfolio_view` 的编辑不等于修改真实资产，只是改变展示和聚合口径。
4. Futu 是 P0 主源，但主源并不等于可自动执行交易。
5. 手工、OCR、消息输入都必须经过确认中心或等价确认流程后才可进入事实层。
6. 自动下单不做，任何页面文案都不能暗示“确认后即自动交易”。
7. Sell Put 页面只能在合规、数据完整且满足纪律的前提下生成草稿。

## 埋点指标

P0 需要围绕“看得见、切得动、确认得清楚”建立指标：

| 指标 | 目的 |
| --- | --- |
| Dashboard 首屏到达率 | 判断用户是否能快速进入持仓主场 |
| `portfolio_view` 切换率 | 判断多视图是否真正被使用 |
| 持仓页下钻率 | 判断摘要是否能驱动进一步查看 |
| 股票 / ETF 详情打开率 | 判断单标的分析价值 |
| Sell Put 草稿进入确认率 | 判断期权工作台是否形成闭环 |
| 确认完成率 | 判断高注意力动作是否顺利处理 |
| 冲突处理完成率 | 判断对账和修正是否可闭环 |
| 数据新鲜度曝光率 | 判断用户是否感知来源和时效 |
| 过期数据下的阻断率 | 判断高风险建议是否被正确拦截 |

## P0 / P1 范围

### P0

1. Supabase Auth 登录后进入 Dashboard。
2. 多 `portfolio_view` 支持。
3. Dashboard 持仓摘要。
4. 持仓工作台。
5. 股票 / ETF 详情。
6. Sell Put 工作台。
7. 确认中心联动。
8. Futu 主源展示。
9. 数据新鲜度、来源与对账状态展示。
10. 不做自动下单。

### P1

1. 更细粒度的持仓筛选和比较。
2. 更完整的历史时间线和复盘入口。
3. 更丰富的期权策略入口。
4. 更深层的分析解释和总结卡片。
5. 更细的移动端交互优化。
6. 与持仓核心不冲突的辅助能力，例如导出、更多排序和批量操作。

## 验收标准

1. 用户登录后能进入 Dashboard，并看到自己的资产摘要。
2. 用户能切换至少两个 `portfolio_view`，且切换后持仓展示随之更新。
3. 持仓工作台能清晰区分股票 / ETF 与期权。
4. 用户能从持仓列表进入股票 / ETF 详情，并看到盈亏、纪律、时间线和策略信息。
5. 用户能从期权入口进入 Sell Put 工作台，并看到现金占用、DTE、IV 和候选合约。
6. 用户提交的交易草稿、OCR 修正、规则 override 和数据冲突都能进入确认中心。
7. 所有需要确认的动作，都不能绕过确认中心直接生效。
8. 页面能明确展示 Futu 主源、新鲜度、来源状态和冲突状态。
9. 数据过期或冲突时，系统能降级展示，不输出高风险建议为既成事实。
10. 页面文案不得暗示自动下单能力。

## 开发前已确认

1. Sell Put 首期仅覆盖 single-leg cash-secured Sell Put；covered call / spread 仅保留数据和导航扩展点，不做完整 UI。
2. Dashboard 顶部资金摘要中，期权市值、现金担保/保证金占用、可用现金需要拆开展示，避免“总资产”误导。
3. `portfolio_view` 首期基于第一期同步到的投资标的范围创建默认视图，覆盖 A 股、港股、美股或期权；用户后续可新增多个 view。
4. 移动端持仓页保留摘要 + 核心筛选，下钻页提供完整字段。

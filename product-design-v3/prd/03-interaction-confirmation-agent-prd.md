# 交互、确认与 Agent 运行体验 PRD

## 背景

AI 持仓系统 3.0 的用户体验不是“一个统一聊天框”，而是围绕 WebApp、微信 claw、确认中心和长任务运行状态形成闭环。OpenClaw 负责高频交互、轻量回复和推送闭环，Hermes 负责复杂深研、长任务和可恢复的异步工作流，WebApp 负责结构化操作、复杂确认、页面内 AI 和账号级管理。

已确认的产品结论如下：

| 决策 | 结论 |
| --- | --- |
| WebApp 全局聊天入口 | 不需要 |
| 微信绑定 / 券商授权 / 账号切换 | 不在微信里做 |
| 绑定与券商授权 | 只在 WebApp / 后台做 |
| 日常文本与意图理解 | 使用 MiniMax M2.7 |
| 图片 / 语音能力 | 通过 Media Tools 处理 |
| 复杂深研 | 交给 Hermes |
| 确认含义 | 确认不等于自动下单授权 |

本 PRD 只讨论交互体验、确认体验、页面内 AI、OpenClaw + Hermes handoff、任务进度、错误降级、通知推送与 P0/P1 范围，不展开底层实现细节。

## 目标

1. 让用户快速判断“现在该看什么、该确认什么、该等待什么”。
2. 让 WebApp 和微信 claw 分工清晰，避免两边各自长出一套产品逻辑。
3. 让确认中心成为所有高注意力动作的统一收口点。
4. 让页面内 AI 只在上下文内工作，不退化成全局聊天。
5. 让长任务有明确状态、进度反馈和补偿机制。
6. 让数据过期、冲突、缺字段、服务失败时的体验统一且安全。

## 非目标

1. 不做 WebApp 独立全局聊天。
2. 不在微信里完成绑定、券商授权或账号切换。
3. 不把确认中心设计成自动下单入口。
4. 不做所有页面同一套通用聊天 UI。
5. 不要求用户理解内部技术状态名，但要能看懂产品状态。
6. 不把 Hermes 变成直接改事实的执行器。

## 用户场景

| 场景 | 用户期望 | 产品结果 |
| --- | --- | --- |
| 微信里问“我现在持仓怎么样” | 快速知道资产、风险和待办 | 返回简洁状态型回复 |
| 在股票详情页问“这只要不要继续拿” | 得到页面内分析和下一步建议 | 给出上下文内分析，不离开页面 |
| 在 Sell Put 页筛选候选 | 快速比较现金占用、到期和风险 | 若条件满足，可生成草稿并进入确认 |
| 录入交易或修正 OCR | 确认自己到底改了什么 | 进入确认中心，不直接落事实 |
| 发起深研 | 先知道已接收，再知道何时更新 | 进入 Hermes，并可查询进度 |
| 数据源或对账异常 | 不误把不可靠结果当成可执行建议 | 进入降级或阻断状态 |

## 交互入口

### WebApp 入口

| 入口 | 典型动作 | 允许的 AI 形态 |
| --- | --- | --- |
| Dashboard | 解释今日变化、生成摘要、查看待办 | 轻量解释 |
| 持仓工作台 | 识别当前视图、下钻持仓、查看风险 | 上下文分析 |
| 股票 / ETF 详情 | 分析持仓、解释波动、生成止盈止损建议 | 页面内 AI |
| Sell Put 工作台 | 比较候选、解释风险、生成草稿 | 页面内 AI + 草稿 |
| 确认中心 | 查看待确认对象、确认 / 拒绝 / 退回 | 结构化审阅，不做泛聊 |
| 任务中心 / 报告页 | 查看 Hermes 结果和历史 | 结果解释 |

### 微信 claw 入口

| 入口 | 典型动作 | 允许的体验 |
| --- | --- | --- |
| 日常对话 | 查持仓、查状态、查任务 | 简短回复 |
| 确认卡片 | 确认 / 修改 / 放弃 | 快确认、快撤回 |
| 任务推送 | 进度、完成、失败、取消 | 结果通知 |
| 深链跳转 | 复杂编辑或深确认 | 跳 WebApp |

## 确认中心 PRD

确认中心是所有高注意力动作的统一收口页。凡是会改变事实、产生草稿、修正规则、处理冲突或批量导入的内容，都必须先进入确认中心。

### 覆盖对象

| 类型 | 示例 | 用户应看到的核心信息 |
| --- | --- | --- |
| 交易事实 | 手工买入 / 卖出 / 开平仓 | 标的、数量、价格、时间、来源 |
| OCR 修正 | 截图识别结果修正 | 原始结果、修正结果、置信度 |
| 交易草稿 | Sell Put、roll、加仓、平仓草稿 | 风险摘要、前提条件、动作上限 |
| 规则 override | 纪律例外 | 违反了什么、为何允许、谁确认 |
| 批量导入 | CSV、对账单导入 | 范围、数量、冲突项 |
| 数据冲突 | 券商 / 手工 / OCR 不一致 | 冲突来源、建议处理方式 |

### 状态与文案边界

| 状态 | 用户可见文案方向 | 边界 |
| --- | --- | --- |
| 待确认 | “请先确认这份结构化动作” | 说明对象，不催促下单 |
| 已确认 | “已记录 / 已提交确认动作” | 只代表进入后续流程 |
| 已拒绝 | “已放弃这次动作” | 不继续写入事实层 |
| 已过期 | “当前版本已失效，请重新确认” | 旧草稿不能复用 |
| 处理中 | “正在提交，请稍等” | 与最终成功分开 |

### 关键规则

1. 确认中心确认的是“规范化动作”，不是原始聊天话术。
2. 确认不等于自动下单授权；任何文案都不能暗示“确认后会自动交易”。
3. 微信和 WebApp 读取同一确认事实源，不能各自维护真相。
4. 确认对象必须展示来源、数据时点、风险等级和动作上限。
5. 确认后如果提交失败，要保留可恢复状态，不得伪装成成功。

### 体验要求

| 体验点 | 要求 |
| --- | --- |
| 结构化预览 | 先让用户看清对象、来源、风险和影响范围 |
| 显式决策 | 只保留确认、拒绝、退回三类主操作 |
| 审计可追溯 | 保留确认人、时间、渠道和源快照 |
| 双端一致 | 微信上确认后，WebApp 立即同步状态 |
| 复杂场景分流 | 字段太多、理由太长、风险太高时跳 WebApp |

## 页面内 AI PRD

WebApp 不做全局聊天入口，AI 只能嵌入具体页面上下文中，以“解释、分析、建议、发起任务、生成草稿”为主。

### 页面级 AI 形态

| 页面 | AI 入口 | 允许输出 |
| --- | --- | --- |
| Dashboard | 解释今日变化、生成摘要 | `info_only` / 轻量解释 |
| 持仓工作台 | 解释当前视图、分析持仓结构 | `analysis_only` |
| 股票 / ETF 详情 | 分析持仓、生成止盈止损建议 | `analysis_only` / `suggested_action` |
| Sell Put 工作台 | 比较候选、解释风险、生成草稿 | `analysis_only` / `trade_draft` |
| 规则页 | 检查纪律、解释冲突 | `analysis_only` |
| 确认中心 | 解释差异、辅助审阅 | 仅辅助，不生成新建议 |

### 文案边界

1. 页面内 AI 必须带页面上下文、数据时点和来源提示。
2. 不能把“建议观察”包装成“可以下单”。
3. 当数据过期、对账失败或规则不可用时，页面内 AI 只能降级为解释型输出。
4. 日常文本和意图由 MiniMax M2.7 处理，图片 / 语音通过 Media Tools 处理，复杂深研交给 Hermes。

### 交互原则

1. AI 入口必须贴在具体页面动作旁边，而不是漂浮在全局角落。
2. 每个入口都要有明确的动作上限。
3. 复杂任务要能一键发起 Hermes，不在页面里硬拖长对话。
4. 页面内 AI 输出可以帮助用户生成草稿，但不能替代确认中心。

## 微信与 WebApp 分工

| 能力 | 微信 claw | WebApp |
| --- | --- | --- |
| 日常查询 | 主入口 | 结构化补充 |
| 深度分析 | 接收摘要和结果 | 主入口之一 |
| 确认交互 | 快确认、快放弃 | 深确认主场 |
| 绑定 / 授权 | 不做 | 主入口 |
| 账号切换 | 不做 | 主入口 |
| 页面内 AI | 不承载复杂页面上下文 | 主入口 |
| 任务推送 | 主投递渠道 | 消息中心补充 |

微信负责快、轻、短，WebApp 负责深、准、可审计。任何需要完整表单、字段编辑、规则说明或复杂确认的动作，都应该导向 WebApp。

## 长任务 / Hermes handoff

### 什么时候 handoff

| 请求类型 | 去向 |
| --- | --- |
| 个股深研 | Hermes |
| Sell Put 深度筛选 | Hermes |
| 多步复盘 / 年度归因 | Hermes |
| 需要等待外部数据且周期较长的任务 | Hermes |

### 用户可见状态

| 状态 | 用户理解 |
| --- | --- |
| 已接收 | 系统已经收到了 |
| 排队中 | 还没开始跑 |
| 分析中 | Hermes 正在工作 |
| 等待数据 | 在等行情 / 券商 / 外部工具 |
| 已有阶段结果 | 先能看部分内容 |
| 已完成 | 结果已就绪 |
| 已失败 | 这次没跑通 |
| 已取消 | 用户或系统取消了 |

### 体验要求

1. OpenClaw 收到深任务后，要先给用户一个“已接收 / 排队中”的明确反馈。
2. 任务状态查询必须能回答“现在到哪一步了、最近一次更新时间是什么、是否在等外部数据”。
3. Hermes 只能做复杂分析和结果沉淀，不能直接修改事实层或绕过确认。
4. 长任务要支持进度补偿：推送失败后，结果仍应可在消息中心或任务页查看。
5. 任务取消和失败必须保留清晰状态，不允许静默消失。

## 错误与降级

### 基本原则

错误不是只返回“失败”，而是要告诉用户现在还能做什么、不能做什么、何时可以恢复。

### 降级场景

| 场景 | 产品反应 |
| --- | --- |
| 数据过期 | 明确标注时点，只能做观察型输出 |
| 券商对账失败 | 不输出高置信持仓或可执行 sell put 建议 |
| 期权关键字段缺失 | 阻断候选输出，只保留解释 |
| 规则服务不可用 | 高风险动作阻断，低风险查询可继续 |
| Hermes 不可用 | 轻量查询继续，长任务排队或稍后重试 |
| OpenClaw / 投递失败 | 结果进入补偿队列，稍后补发 |
| 来源冲突 | 进入确认中心，不直接覆盖 |

### 安全文案边界

| 情况 | 推荐文案方向 |
| --- | --- |
| 可继续观察 | “我可以先做观察分析，但这次不能作为交易级依据” |
| 高风险阻断 | “关键数据暂不可确认，因此不能给出可执行建议” |
| 长任务延迟 | “任务已开始，当前在等待外部数据” |
| 推送补偿 | “结果已生成，正在补发” |

## 通知 / 推送

通知不是“尽可能多发”，而是“只在用户真正需要知道时发”。

### 推送节点

| 节点 | 是否推送 |
| --- | --- |
| 任务已接收 | 是 |
| 长时间等待 | 是 |
| 有阶段性结果 | 是 |
| 任务完成 | 是 |
| 任务失败 | 是 |
| 任务取消 | 是 |
| 普通低价值状态抖动 | 否 |

### 推送要求

1. 推送内容必须短，包含任务名、状态、最近更新时间和下一步。
2. 推送失败后要走补偿，不允许丢失结果。
3. 夜间或静默时段只保留高价值通知。
4. 推送内容必须与 WebApp 和微信确认状态一致。

## P0 / P1 范围

### P0

1. WebApp 不做全局聊天入口。
2. 微信不做绑定、授权和账号切换。
3. Dashboard、持仓工作台、股票 / ETF 详情、Sell Put 工作台、确认中心都具备页面内 AI 入口。
4. 确认中心统一承接交易事实、OCR 修正、规则 override、草稿确认和冲突处理。
5. Hermes 长任务具备接收、排队、进度查询和完成推送。
6. 关键错误场景具备降级文案和状态回写。
7. 通知 / 推送支持补偿和状态一致性。

### P1

1. 更丰富的 OCR 多字段修正。
2. 语音入口和语音摘要的完善。
3. 更细的任务进度与阶段回放。
4. 更丰富的推送模板和消息中心。
5. 更多页面内 AI 模板，但仍然不做全局聊天。

## 验收标准

1. WebApp 中不存在独立全局聊天入口。
2. 微信里无法完成绑定、券商授权或账号切换，只能跳转到 WebApp。
3. 任何高注意力动作都能落到确认中心，并显示结构化明细与来源。
4. 确认页面明确写出“确认不等于自动下单授权”。
5. 页面内 AI 只在对应页面上下文中出现，不越界到全局。
6. MiniMax M2.7 只承担日常文本 / 意图类回复，图片 / 语音能力通过 Media Tools 处理。
7. Hermes 长任务能展示接收、排队、分析、等待数据、完成、失败、取消等状态。
8. 数据过期、对账失败、规则不可用或关键字段缺失时，系统不会输出可执行交易建议。
9. 推送失败后，结果可通过补偿恢复，不会静默丢失。
10. 微信和 WebApp 对同一确认对象展示一致状态。

## 开发前已确认

1. 确认中心默认有效期按风险分级：高风险 30 分钟，低风险 24 小时。
2. Sell Put 草稿在 P0 允许生成，但不自动下单，必须进入待确认对象。
3. WebApp 消息中心与微信推送共用 `delivery/outbox/message_events` 事实源，微信与 WebApp 只是不同 channel view。
4. 语音输入进入 P0：支持语音输入/语音口令、ASR 识别和二次确认；完整语音输出和多轮语音体验放 P1。
5. 长任务阶段名称统一为平台级词表：`queued`、`collecting`、`analyzing`、`reviewing`、`ready`、`failed`。

# Hermes 微信消息交互产品模块方案

## 1. 设计目标

本方案把 Obsidian 中的 3.0 产品模块重新收敛到 **Hermes 系统的微信消息交互版**。首期不包含 WebApp 页面，不设计移动端/桌面端界面，只定义用户通过微信 claw 与 Hermes Gateway 交互时需要具备的产品模块、消息形态、写入回执边界和任务流。

核心目标：

1. 用户主要通过微信完成持仓查询、交易记录、关注清单维护、深度分析发起、任务进度查询和提醒接收。
2. 股票/ETF 与期权继续作为两个并行产品域，但用户入口统一为微信对话。
3. 持仓前、持仓中、持仓后继续作为主生命周期，但产品表现从“消息指令 + 写入回执 + 主动推送 + 长任务报告”展开。
4. Hermes 负责深度研究、期权策略、复盘归因和 memory 沉淀；金融事实由 Hermes 产品后端的资产、规则、关注和任务服务写入，并记录来源与审计。本次不引入独立工具层，也不设计二次确认流程。
5. 微信交互必须明确数据时点、来源、可行动等级和写入结果，避免模型把观察分析包装成可执行交易建议。

## 2. 首期产品边界

### 做什么

| 范围 | 首期能力 |
| --- | --- |
| 微信入口 | 文本消息、修改/撤销、任务进度查询、系统主动推送 |
| 持仓查询 | 当前资产摘要、股票/ETF 持仓、期权持仓、现金/保证金、数据质量 |
| 持仓维护 | 自然语言交易记录、手工持仓修正、OCR/截图识别后的写入回执 |
| 关注清单 | 加入关注、设置触发条件、查看关注理由、重新分析 |
| 交易纪律 | 查询规则、设置/修改规则、规则命中提醒、override 原因记录 |
| 股票分析 | 个股快问快答、止盈止损观察、深研任务发起和结果推送 |
| 期权分析 | Sell Put 标的适合性、长桥期权链候选、现金/保证金风险、持仓监控 |
| 持仓后复盘 | 清仓总结、纪律评分、二次买入条件、assignment 复盘 |
| 长任务 | 深研、sell put 扫描、年度/阶段复盘、异常诊断 |

### 不做什么

1. 不做 WebApp 页面、管理后台页面或独立状态处理页面。
2. P0 不做券商授权；账号绑定通过用户注册模块完成，微信内不做绑定、授权或敏感凭证管理。
3. 不自动下单，不生成任何券商执行动作。
4. 不让 LLM 自由文本结果直接覆盖 `portfolio_positions`、`trade_events`、`trading_rules` 等金融事实；写入必须先结构化解析，并记录来源、回执和审计。
5. 不把期权风险压扁成普通盈亏；Sell Put 必须单独经过期权规则、现金/保证金和风险门。

## 3. 产品拆分

微信-only 版本仍拆成两个产品域，共享账号、数据、推送、memory、审计和任务基础设施。

| 产品域 | 用户在微信里的核心问题 | 微信能力 |
| --- | --- | --- |
| 股票/ETF 持仓产品 | 我现在持有什么，风险在哪，什么时候该观察/复盘/调整 | 持仓摘要、个股解释、机会关注、止盈止损观察、异动提醒、交易记录、清仓复盘、二次买入 |
| Sell Put 期权产品 | 哪些标的适合卖 put，卖哪个 strike，风险和资金占用如何 | 标的适合性、期权链候选、IV/Greeks、现金/保证金、assignment 风险、到期监控、roll/close/接股预案 |

## 4. 微信交互模块总览

| 模块 | 用户触发方式 | 系统输出 | 写入边界 |
| --- | --- | --- | --- |
| 即时持仓问答 | “现在持仓怎么样”“看下 AAPL” | 短摘要、关键数据、风险、数据来源 | 只读 |
| 交易记录写入 | “买入 AAPL 10 股 180” | 结构化写入回执 | 解析后写 `trade_events` 或手工事实，可修改/撤销 |
| OCR/截图写入 | 发送券商截图、成交截图、持仓截图 | 识别结果 + 低置信字段提示 | 高置信字段写入并回执，低置信字段要求补充 |
| 关注清单维护 | “关注 NVDA，跌到 110 提醒我” | 关注卡片、触发条件、复核时间 | 直接写入 `follow_view_items`，记录来源和审计 |
| 纪律规则维护 | “以后不要买中概股”“单票不超过 20%” | 规则回执、影响说明、修改口令 | 解析后写 `trading_rules`，override 必须带原因 |
| 股票分析 | “AAPL 要不要止盈”“做 NVDA 深研” | 快速观察或 Hermes 长任务报告 | 分析 artifact，不写持仓事实 |
| Sell Put 分析 | “下周适合卖 put 吗”“看 TSLA put” | 标的适合性、候选合约、阻断原因 | 只生成分析/监控/交易草稿 |
| 期权持仓监控 | 系统推送或用户查询 | DTE、delta、IV、保证金、assignment 风险 | 只读或创建处理草案回执 |
| 清仓复盘 | “复盘我清仓的 TSLA” | 复盘摘要、纪律评分、二次买入条件 | 写复盘 artifact；二次买入条件可写入关注清单 |
| 任务与推送 | “进度”“取消任务”“发完整报告” | 任务状态、结果摘要、补推 | 任务状态写入，不改金融事实 |

## 5. 生命周期产品模块

### 5.1 持仓前：关注、机会和策略候选

| 微信场景 | 示例口令 | 后台模块 | 输出 |
| --- | --- | --- | --- |
| 加入关注 | “关注 NVDA，回调到 110 提醒” | `follow_views` / `follow_view_items` | 关注卡片、触发条件、下次复核 |
| 机会捕捉 | “最近有什么值得关注” | Hermes 深研模块 + 市场数据服务 | 候选机会、催化剂、风险标签 |
| 个股预分析 | “帮我看下 AMD 值不值得跟踪” | Equity Analyst | thesis、买点区间、风险点 |
| Sell Put 标的筛选 | “找下周适合 sell put 的标的” | Sell Put 策略模块 + 长桥期权链 | 标的适合性、阻断原因、候选池 |
| 策略纪律检查 | “这个标的符合我的规则吗” | 交易纪律服务 | pass/warn/block 与原因 |

持仓前模块的核心不是直接推荐交易，而是形成 **可追踪的候选对象**。微信输出应默认落为：

```text
结论：是否值得关注
依据：行情/基本面/事件/规则
风险：数据缺失、纪律冲突、事件窗口
下一步：加入关注 / 发起深研 / 设置提醒 / 放弃
来源：数据源与时间
```

### 5.2 持仓中：当前资产、风险和操作回执

| 微信场景 | 示例口令 | 后台模块 | 输出 |
| --- | --- | --- | --- |
| 当前资产摘要 | “现在持仓怎么样” | `portfolio_views` read model | 总资产、现金、股票/ETF、期权、主要风险 |
| 单标的详情 | “看下 AAPL” | Equity/Option read model | 数量、成本、现价、盈亏、仓位、数据质量 |
| 止盈止损观察 | “AAPL 现在要不要止盈” | 股票分析模块 + 风险分级服务 | 观察价、触发条件、纪律命中 |
| 异动提醒 | 系统主动推送 | 市场数据服务 + 推送服务 | 异动原因、影响范围、建议等级 |
| 交易录入 | “卖出 TSLA 5 股 250” | 资产写入服务 | 交易写入回执 |
| 期权持仓监控 | “我的 short put 风险怎么样” | 期权分析模块 + 长桥期权链 | DTE、delta、IV、cash secured、assignment |
| Roll/Close/接股预案 | “TSLA put 快到期怎么办” | Sell Put 策略模块 | 情景方案、阻断条件、草稿上限 |

持仓中模块必须区分 `info_only`、`analysis_only`、`suggested_action`、`trade_draft`、`blocked`。例如现金/保证金不可验证时，Sell Put 回复必须降级为 `blocked` 或 `analysis_only`，不能输出“可卖某个 strike”。

### 5.3 持仓后：复盘、经验和二次买入

| 微信场景 | 示例口令 | 后台模块 | 输出 |
| --- | --- | --- | --- |
| 清仓复盘 | “复盘我这次清仓 TSLA” | `list_views` + Hermes Review | 买卖点、收益贡献、纪律评分 |
| 二次买入 | “TSLA 什么时候可以重新关注” | 股票分析模块 + 关注清单服务 | rebuy conditions、触发提醒、关注卡片 |
| 纪律归因 | “我今年哪里违反纪律最多” | Discipline Checks + Hermes Review | 违规类型、影响、改进建议 |
| Assignment 复盘 | “这次被指派做个复盘” | Options Review | 接股成本、后续表现、规则改进 |

持仓后模块的写入对象主要是复盘 artifact、`list_view_items.review_summary`、`rebuy_conditions` 和策略经验 memory。研究结论不能自动写成持仓事实或硬性交易规则；如果要沉淀为规则，必须由用户用明确口令表达规则内容和原因。

## 6. 微信消息形态

### 6.1 状态型回复

用于回答“现在是什么状态”。默认只读，必须带数据时点。

```text
AAPL 当前持仓：浮盈 +12.4%，仓位 18.2%

关键数据：
- 数量：100 股
- 成本：180.20
- 现价：202.55
- 数据：Longbridge / 系统资产视图，09:31:20

风险：
- 仓位接近你设置的单票 20% 上限
- 下周有财报，止盈建议只能作为观察

可操作：
- 回复“止盈计划 AAPL”
- 回复“设提醒 AAPL 195”
```

### 6.2 分析型回复

用于回答“为什么、风险在哪里、下一步关注什么”。若数据源降级、行情过期或规则服务不可用，只能输出观察分析。

固定结构：

```text
结论
关键依据
主要风险
行动等级
可继续口令
来源与时间
```

### 6.3 操作型回复

用于改变系统状态，例如交易记录、规则、关注清单、提醒、任务取消。操作型回复必须告诉用户“这会写入什么，不会做什么”。

示例：

```text
已记录一条交易：

动作：记录买入
标的：AAPL
数量：10 股
价格：180.00
账户：美股账户
来源：微信文本

这只写入系统交易记录，不会自动下单。

回复：
修改 AAPL 10 股 181
撤销刚才记录
```

### 6.4 写入回执与修正回复

本次不设计二次确认流程。写入型消息采用“解析后写入 + 结构化回执 + 可修改/撤销”的方式。最少字段：

| 字段 | 说明 |
| --- | --- |
| 动作类型 | 交易录入、OCR 修正、规则更新、规则 override、交易草稿 |
| 账号 | 用户可读账号名 + 内部 `tenant_id` 短码 |
| 关键事实 | 标的、数量、价格、金额、规则表达式、候选合约 |
| 数据来源 | 文本、截图、券商消息、OCR、Hermes artifact |
| 风险提示 | 命中的规则、缺失数据、是否只记录不下单 |
| 可选回复 | 修改、撤销、查看详情、补充字段 |
| 回执编号 | 便于用户引用刚才的写入结果 |

写入回执语义：

1. 用户发出明确操作口令后，系统按解析结果写入并立即回执。
2. 回执只代表系统记录、草稿、规则或 artifact 已更新，不代表自动下单。
3. 用户可用“修改刚才记录”“撤销刚才记录”“补充原因”等口令修正。
4. 低置信 OCR、字段缺失、账号上下文不明确时不写关键事实，只要求用户补充。

### 6.5 主动推送

主动推送必须来自 `delivery_outbox`，不能边生成边发送。

| 推送类型 | 触发 | 内容 |
| --- | --- | --- |
| 每日收盘摘要 | 市场收盘后 | 资产变化、主要风险、待处理事项 |
| 持仓异动 | 价格/新闻/财报/成交 | 异动原因、影响仓位、数据时点 |
| 止盈止损提醒 | 规则触发 | 观察价、建议等级、后续口令 |
| 期权风险提醒 | DTE、delta、接近 strike、保证金变化 | assignment 风险、roll/close/接股观察 |
| 深研完成 | Hermes job 完成 | 摘要、结论等级、来源、后续口令 |
| 系统异常 | 资产数据刷新失败、长桥数据源故障 | 影响范围、当前降级策略 |

## 7. 意图分类与路由

| Intent | 示例 | Runtime | Actionability |
| --- | --- | --- | --- |
| `portfolio_query` | “现在持仓怎么样” | 微信快速查询模块 | `info_only` |
| `position_detail` | “看下 AAPL” | OpenClaw-side，必要时 Hermes | `info_only` / `analysis_only` |
| `trade_record_input` | “买入 TSLA 5 股” | 资产写入服务 | `trade_draft` |
| `ocr_position_import` | 发送截图 | OCR/Vision + 资产写入服务 | `trade_draft` / `blocked` |
| `follow_add` | “关注 NVDA” | 关注清单服务 | `suggested_action` |
| `rule_update` | “以后不要买中概股” | 规则写入服务 | `trade_draft` |
| `stop_take_profit` | “AAPL 止盈计划” | 股票分析模块 + 风险分级服务 | `analysis_only` / `suggested_action` |
| `sell_put_analysis` | “这个适合卖 put 吗” | Hermes 期权策略模块 | `analysis_only` / `suggested_action` / `trade_draft` |
| `option_monitor` | “short put 风险怎么样” | 期权分析模块 | `info_only` / `analysis_only` |
| `deep_research` | “做个 NVDA 深研” | Hermes 深研模块 | artifact |
| `closed_review` | “复盘清仓 TSLA” | Hermes 复盘模块 | artifact |
| `task_progress` | “进度” | 任务状态服务 | `info_only` |
| `task_cancel` | “取消任务” | 任务状态服务 | controlled write |
| `delivery_issue` | “昨天没收到日报” | 推送/运维状态服务 | `info_only` |

路由原则：

1. 快速查询不进入 Hermes 长任务，优先同步回复。
2. 深研、Sell Put 扫描、清仓复盘、复杂归因进入 Hermes job。
3. 高风险写入和交易草稿必须输出结构化回执、风险提示和可撤销/可修改口令。
4. 本次不引入独立工具层；路由只区分微信快速处理、Hermes 长任务和产品后端写入服务。

## 8. Sell Put 微信产品细化

首期 Sell Put 只覆盖 single-leg cash secured put / short put，不扩展复杂组合。

### 8.1 标的适合性

用户问：

```text
TSLA 适合卖 put 吗
```

系统先回答标的能不能做 Sell Put，而不是直接给 strike：

| 检查项 | 说明 |
| --- | --- |
| 用户是否愿意接股 | 不愿接股则不生成交易草稿 |
| 现金/保证金是否来自系统资产视图且可验证 | 不可验证则 blocked |
| 标的行情是否 fresh | 主源失败则降级 |
| 财报/事件窗口 | 默认不跨财报生成执行清单 |
| 纪律规则 | 黑名单、集中度、DTE、delta、现金占用 |
| 基本面与趋势 | 被指派后是否可接受持有 |

### 8.2 候选合约

只有标的适合性通过后，才进入长桥期权链候选。微信输出最多展示 1-3 个候选，完整排序进入 artifact。

```text
TSLA Sell Put：当前只建议观察，不生成交易草稿

阻断原因：
- 下次财报在到期窗口内
- 可用现金未完成系统资产视图校验

可继续：
- 回复“只看观察候选”
- 回复“设置 TSLA put 监控”
- 回复“现金刷新后重算”
```

若满足草稿条件：

```text
候选 1：TSLA 2026-07-17 Put 180

关键数据：
- DTE：37
- Delta：-0.22
- 权利金中位：3.20
- Breakeven：176.80
- Cash secured：18,000

风险：
- 指派后 TSLA 仓位将到 14%，超过你的 12% 偏好，已标记为需要人工判断
- 不自动下单，只生成系统交易草稿

回复：
生成草稿
加入监控
放弃
```

## 9. 长任务体验

Hermes 长任务不让用户在微信里等待。

| 阶段 | 微信体验 |
| --- | --- |
| 创建任务 | “已开始 NVDA 深研，预计 5-10 分钟。可回复‘进度’。” |
| 进行中 | “正在分析财报、历史波动和期权链。” |
| 失败/降级 | “长桥期权链缺少 IV/Greeks，已降级为股票观察分析。” |
| 完成 | 摘要 + 结论等级 + 数据来源 + 后续口令 |
| 取消 | 校验任务归属后取消，已进入不可取消阶段则说明原因 |

常用口令：

| 口令 | 动作 |
| --- | --- |
| “进度” | 查看最近一个 Hermes 任务 |
| “取消任务” | 取消最近一个可取消任务 |
| “继续分析” | 恢复暂停/失败可继续的任务 |
| “发完整报告” | P0 推送微信长摘要；完整 artifact 先存储在后台 |
| “今天还有什么待处理” | 列出待处理事项 |

## 10. 数据与写入边界

### 10.1 事实来源优先级

| 来源 | 用途 | 写入条件 |
| --- | --- | --- |
| 系统账户资产视图 | 持仓、成交、现金、保证金的 P0 主事实视图 | 来自注册账号下的系统记录、用户消息写入和后续刷新结果 |
| Longbridge 市场数据 | 美股/港股期权链、行情、IV/Greeks 等市场数据 | 只读数据源，不能替代用户资产事实 |
| 微信文本 | 用户手工交易、规则、关注条件 | 解析后写入并回执；关注清单条件可直接写 |
| 微信截图/OCR | 交易通知、成交、持仓截图 | 高置信字段写入并回执，低置信字段要求补充 |
| Hermes artifact | 深研、复盘、策略建议 | 写 artifact，不写持仓事实 |
| Memory | 用户偏好、经验、复盘教训 | 必须 tenant scoped，不能作为资产事实 |

### 10.2 写入权限

| 写入对象 | 是否允许微信触发 | 写入方式 |
| --- | --- | --- |
| `trade_events` | 允许 | 解析后写入并回执，支持修改/撤销 |
| `manual_positions` / 手工修正 | 允许 | 解析后写入并回执，低置信字段要求补充 |
| `follow_view_items` | 允许 | 直接写入，包含关注标的、买入区间、sell put 条件、触发提醒和规则影响 |
| `trading_rules` | 允许 | 解析后写入并回执，override 必须记录原因 |
| `discipline_checks` | 系统生成 | 自动生成，但需可追溯 |
| `research_artifacts` | Hermes 生成 | 自动写入，但不能伪装成事实 |
| `delivery_outbox` | 系统生成 | 不需要 |
| 下单执行 | 不允许 | 不在本范围 |

## 11. P0 / P1 范围建议

### P0：微信可用闭环

1. 文本持仓查询：当前资产、单标的、现金/保证金、期权摘要。
2. 自然语言交易记录：解析、写入、回执、可修改/撤销、审计。
3. 关注清单：加入关注、触发条件、复核提醒。
4. 交易规则：创建/修改规则，写入后回执，override 必须记录原因。
5. Sell Put 单标的分析：标的适合性 + 长桥候选合约 + 阻断原因。
6. Hermes 长任务：深研、Sell Put 扫描、清仓复盘，支持进度查询和结果推送。
7. 主动推送：每日摘要、持仓异动、期权 DTE/assignment 风险、待处理提醒。
8. 写入回执：修改、撤销、补充字段、幂等、审计。
9. 降级回复：数据过期、长桥数据源失败、期权链缺字段、规则服务不可用。

### P1：体验增强

1. 微信图片/OCR 导入持仓和成交截图。
2. 语音转文本后进入同一写入回执流。
3. 链接/转发新闻进入深研材料池。
4. 期权组合监控：到期梯队、roll/close/接股预案。
5. 复盘自动沉淀 memory，但必须区分事实、偏好、经验。
6. 图表/语音推送：由专用媒体服务生成，不让模型自由生成金融图表。
7. 任务失败重试、补推、用户 quiet hours。

## 12. 已确定产品边界

以下口径已确定，作为 P0 产品边界：

1. **没有 WebApp 的首期，账号绑定和券商授权放在哪里完成？**  
   P0 没有券商授权；账号绑定通过用户注册模块完成，且这部分已经在开发中。微信内不做账号绑定，也不做券商授权。

2. **首期是否允许微信创建 `follow_view_items` 时不二次确认？**  
   允许。关注某标的、买入区间、sell put 条件、触发提醒、规则影响都不需要二次确认，系统直接写入并回执，但必须记录来源、解析结果和审计。

3. **交易记录写入是只做“系统记录”，还是要生成“券商执行清单”？**  
   P0 只写系统交易记录和交易草稿，不生成自动下单，也不直接对接券商下单。

4. **Sell Put 首期是否只支持美股/港股期权，还是也要考虑 A 股相关衍生品？**  
   P0 只做美股/港股可从长桥读到期权链的标的；A 股先只做股票/ETF 持仓与关注。

5. **每日摘要推送的默认频率是什么？**  
   按市场收盘后推送一次；盘中只推高优先级异动和待处理提醒。

6. **微信里是否需要多账号切换口令？**  
   不需要多账号切换口令。P0 微信交互只面向当前绑定账号上下文。

7. **Hermes 深研结果是否需要“完整报告链接”？**  
   P0 先推送微信长摘要；完整 artifact 存储在后台，后续再补链接或文件投递。

8. **规则 override 是否允许微信内处理？**  
   允许，但必须要求用户给出原因，并记录到 `discipline_checks` / audit；hard block 类型规则默认不允许微信 override。

## 13. 验收口径

首期微信-only 产品方案可进入研发前，应满足：

1. 任一微信回复都能标注 actionability：只读、观察、建议、草稿或阻断。
2. 任一金融事实写入都能追溯原始消息、结构化解析结果、写入回执和 `tenant_id`。
3. 股票/ETF 与期权在数据、分析和风险上分开处理。
4. Sell Put 在现金/保证金、长桥期权链、事件窗口、规则检查任一关键数据缺失时不能生成可执行草稿。
5. Hermes 只产出分析、artifact、proposal、memory 摘要和 delivery outbox，不直接改写持仓事实。
6. 微信写入语义简单：用户发出操作口令后系统写入并回执，用户可回复“修改/撤销/补充”。
7. 数据源失败时明确降级，不输出高置信行动建议。
